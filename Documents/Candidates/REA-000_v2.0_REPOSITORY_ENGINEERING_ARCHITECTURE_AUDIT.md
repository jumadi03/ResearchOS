## Architecture risk notes

1. `app.main`, routers, and runtime composition are legitimate places for
   outward dependency, so simple cross-layer-import counts must not become a
   blocking law without context.
2. Broad scientific persistence and service facades present concentration risk,
   but immediate splitting could create transaction, schema, and compatibility
   failures.
3. The implemented architecture pipeline is more mature and specific than the
   candidate describes. REA should govern how that implementation is used and
   evolved, not replace its contracts with parallel terminology.
4. Existing production evidence must remain dated evidence. It is not proof
   that the current branch has been deployed or production-accepted.

## Audit conclusion

The repository provides a credible implementation base for a ResearchOS
engineering architecture, especially in deterministic architecture analysis,
testing, deployment assets, monitoring, recovery, and kernel contracts.
Canonical REA publication remains blocked until v2.1 resolves the high-severity
semantic and publication gaps and receives explicit human review.
